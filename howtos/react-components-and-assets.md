# React components and assets

This guide describes how and where to create React components in the `camunda-docs` repository, how and where to place static assets such as images, and how both can be reused across the documentation.

## React components

There are three locations for React components, each serving a different purpose.

### `src/components/`

Use this location for **page-level or feature-level React components** that are embedded directly into MDX documentation pages. These are custom UI elements that are specific to a documentation topic and are not part of the Docusaurus theme.

Examples: `CamundaSelfManaged`, `HelmInstallOverviewMethods`, `Versions`

**Conventions:**

- Create a subdirectory named after the component (for example, `src/components/MyFeature/`).
- Provide an `index.jsx` entry point that exports the main component.
- Place related sub-components in the same directory.
- Co-locate component-specific assets (icons, images, CSS) inside the component's directory (for example, `src/components/MyFeature/icons/`).
- Import the component in MDX files using the `@site` alias:
  ```js
  import MyFeature from "@site/src/components/MyFeature";
  ```

### `src/mdx/`

Use this location for **small, reusable inline components** that are used frequently in MDX content across many documentation pages. These are utility components for enriching prose, for example icons or markers.

Examples: `GitHubInlineIcon`, `MarkerCamundaExtension`, `FormViewer`

**Conventions:**

- Use a single file per component (for example, `src/mdx/MyIcon.jsx`).
- Keep these components small and focused on a single presentational concern.
- Import them in MDX files using the `@site` alias:
  ```js
  import MyIcon from "@site/src/mdx/MyIcon";
  ```

### `src/theme/`

Use this location for **Docusaurus theme customizations** created by [swizzling](https://docusaurus.io/docs/swizzling). These override the default Docusaurus theme components.

Examples: `DocCard`, `DocItem/Metadata`, `Footer`

**Conventions:**

- Components here are generated by the `docusaurus swizzle` command; do not create them manually.
- Each swizzled component must include a comment at the top that answers:
  1. Why is the component swizzled?
  2. Which Docusaurus version was it swizzled from?
- See `src/theme/README.md` for details.

### Components inside `docs/` or `versioned_docs/`

Placing React components directly inside the `docs/` or `versioned_docs/` tree is **discouraged**, because it prevents reuse and requires duplicating the component across versioned copies of the documentation.

An exception exists for components that are tightly scoped to a single doc page and are not expected to be reused. Even in this case, prefer promoting the component to `src/components/` if there is any chance it will be needed elsewhere.

## Assets

### Static assets — `static/img/`

Place images that **do not change between documentation versions** in `static/img/`. These assets are served at `/img/<filename>` and are shared across all versions of the documentation.

Examples: BPMN element icons, form icons.

### Versioned assets — `docs/.../img/`

Place images that **may change from one product version to the next** in an `img/` subdirectory next to the markdown file that uses them. This ensures that the correct image is shown for each version of the documentation.

**Convention:** Add an `img/` directory to each docs directory where version-specific images are needed.

### BPMN and DMN files — `static/bpmn/`

Place `.bpmn` and `.dmn` files in `static/bpmn/`, organized in subdirectories that mirror the documentation structure.

Example: A BPMN file for the **APIs & tools** section goes in `static/bpmn/apis-tools/`.

Reference BPMN files in docs as `/bpmn/<subfolder>/<filename>.bpmn`.

### Component-specific assets

Icons, images, and CSS that belong **exclusively** to a single React component in `src/components/` should be co-located with that component in its own subdirectory (for example, `src/components/MyFeature/icons/`). Do not place these in `static/img/`.

### Source and pre-source files

| Type | Location |
| ---- | -------- |
| Source files for images and videos (for example, Figma exports, raw SVGs) | `media-src/<path matching docs structure>/` |
| Files used to generate source files (for example, BPMN files used to produce images) | `media-pre-src/` |

## Reuse

### Reusing components

- Components in `src/components/` and `src/mdx/` can be imported in any MDX file across all versions of the documentation.
- Use the `@site` alias to create stable import paths that are not affected by the location of the importing file:
  ```js
  import MyComponent from "@site/src/components/MyComponent";
  import MyIcon from "@site/src/mdx/MyIcon";
  ```
- When the same component is needed in both the `docs/` (next) and `versioned_docs/` trees, always keep a **single copy** in `src/components/` or `src/mdx/` and import it from both locations.

### Reusing assets

- Images in `static/img/` can be referenced from any documentation page using an absolute path: `![Alt text](/img/filename.png)`.
- Versioned images in `docs/.../img/` are referenced with a relative path from the markdown file that uses them: `![Alt text](img/filename.png)`.
- To share an image across multiple pages within the same version, consider placing it in the nearest common ancestor `img/` directory or in `static/img/` if it applies to all versions.

## Summary

| Purpose | Location |
| ------- | -------- |
| Page-level/feature-level components | `src/components/<ComponentName>/` |
| Inline MDX utility components | `src/mdx/` |
| Docusaurus theme overrides | `src/theme/` |
| Version-independent static images | `static/img/` |
| Version-specific images | `docs/<path>/img/` or `versioned_docs/<version>/<path>/img/` |
| BPMN/DMN files | `static/bpmn/<path>/` |
| Component-specific icons/images | `src/components/<ComponentName>/icons/` (or similar) |
| Image/video source files | `media-src/<path>/` |
| Pre-source files | `media-pre-src/` |
