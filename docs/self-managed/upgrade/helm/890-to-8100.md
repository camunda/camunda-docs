Upgrade a Helm-managed Camunda 8 Self-Managed deployment from version 8.9 to 8.10.

:::info Upgrade procedure
All Camunda 8 upgrades must follow the required upgrade procedure: upgrade one minor version at a time and do not skip minors. For best stability and fix coverage, use the latest available patch in each minor before and after the minor upgrade.

See [version compatibility checks](../../components/orchestration-cluster/core-settings/concepts/version-compatibility.md#required-upgrade-procedure).
:::

## Prerequisites

Before upgrading, ensure you have met the following prerequisites:

| Prerequisite                | Description                                                                                                                                                                                                                                                                                                                   |
| :-------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Confirm `8.9.x` baseline    | Confirm your deployment is running an `8.9.x` version before upgrading to 8.10. If you are upgrading from a version earlier than 8.9, see [Upgrading from an earlier version](/self-managed/upgrade/index.md#upgrading-from-an-earlier-version). For best stability and fix coverage, use the latest available `8.9.x` patch. |
| Confirm upgrade eligibility | Review [Prepare for upgrade](/self-managed/upgrade/prepare-for-upgrade.md) to confirm upgrade eligibility and complete any required pre-upgrade actions for 8.10.                                                                                                                                                             |
| Confirm Helm chart support  | <p>Review the Camunda Helm chart [version matrix](https://helm.camunda.io/camunda-platform/version-matrix/) and confirm:</p><p><ul><li>The Helm chart version deploying 8.10 is supported for your Kubernetes version.</li><li>Your Helm client version is supported for that chart.</li></ul></p>                            |
| Backups                     | Create and verify backups for your Camunda data. See [Backup and restore](../../../self-managed/operational-guides/backup-restore/backup-and-restore.md).                                                                                                                                                                     |
| Audit `values.yaml`         | Delete every entry listed under [Removed keys](#removed-keys) from your `values.yaml`. | Migrate off bundled Bitnami | If you used the bundled `identityKeycloak`, `identityPostgresql`, `webModelerPostgresql`, or `elasticsearch` subcharts, stand up external replacements and migrate data **before** upgrading. | Test upgrade                | Test the upgrade in a non-production environment using a copy of your production configuration.                                                                                                                                                                                                                               |

## Create your 8.10 values file

Use your existing 8.9 configuration as the starting point for the upgrade.

1. Copy your current `values.yaml` and name the file `values-8.10.yaml`.

2. Download the default values for the Camunda 8.10 Helm chart:

   ```bash
   helm repo update
   helm show values camunda/camunda-platform --version <CHART_VERSION> > values-8.10-default.yaml
   ```

   To identify the latest available chart version, run:

   ```bash
   helm search repo camunda/camunda-platform --versions
   ```

3. Compare `values-8.10.yaml` with `values-8.10-default.yaml` to identify new configuration options, changed defaults, and deprecated or removed settings.

4. Update `values-8.10.yaml` using the configuration tables below before running `helm upgrade`.

## Update your values file to 8.10

Use the following sections to update `values-8.10.yaml` for the 8.9 to 8.10 upgrade.

### Configuration changes

The following sections list configuration keys that changed between chart 14.x (8.9) and 15.x (8.10). If your `values.yaml` uses any of these keys, update them before upgrading.

#### Camunda Hub consolidation

In 8.10, Console and Web Modeler are consolidated into a single component called **Camunda Hub**. Console is no longer deployed as a standalone service; its UI and APIs are served by the new `hub-restapi` container (the renamed Web Modeler REST API). Setting `camundaHub.enabled: true` activates both the Web Modeler and Console feature sets inside this single component.

The legacy top-level keys `webModeler.enabled` and `console.enabled` continue to work as a compatibility shim:

- `webModeler.enabled: true` is equivalent to enabling the Web Modeler feature of Camunda Hub.
- `console.enabled: true` is equivalent to enabling the Console feature of Camunda Hub. It no longer provisions a separate Console deployment; instead, it switches on the Console UI inside the `hub-restapi` pod.

Helm prints a deprecation warning during upgrade if you use the legacy keys. The migration path is the same in either case (the chart maps them to identical runtime configuration), so you can adopt `camundaHub.enabled: true` at your convenience.

**Recommended structure for 8.10:**

```yaml
camundaHub:
  enabled: true
  webModeler:
    restapi:
      mail:
        fromAddress: no-reply@example.com
```

**Legacy structure (still supported with deprecation warnings):**

```yaml
console:
  enabled: true
webModeler:
  enabled: true
  restapi:
    mail:
      fromAddress: no-reply@example.com
```

:::note Resource sizing
Because Console now runs inside the `hub-restapi` pod, you may need to raise the CPU and memory limits you previously set on `webModeler.restapi.resources` to accommodate both workloads. Monitor the pod after upgrade and adjust if you see throttling or OOM events.
:::

#### Image repository changes

Web Modeler Docker images have been renamed in 8.10:

| **8.9 image**                          | **8.10 image**                  |
| :------------------------------------- | :------------------------------ |
| `camunda/web-modeler-restapi:8.9.x`    | `camunda/hub:8.10.x`            |
| `camunda/web-modeler-websockets:8.9.x` | `camunda/hub-websockets:8.10.x` |

When image repositories are not pinned (the default), no action is required — the chart picks up the new names automatically.

When repositories are pinned (typical in air-gapped or mirrored-registry setups), update them:

```yaml
# Before (8.9)
webModeler:
  restapi:
    image:
      repository: my-registry/camunda/web-modeler-restapi
  websockets:
    image:
      repository: my-registry/camunda/web-modeler-websockets

# After (8.10)
webModeler:
  restapi:
    image:
      repository: my-registry/camunda/hub
  websockets:
    image:
      repository: my-registry/camunda/hub-websockets
```

#### Removed keys

The following keys have been removed in 8.10. If any of these are present in your `values.yaml`, **delete them before upgrading**. The chart enforces a hard constraint that will cause the upgrade to fail with an error if these keys are present.

| **Removed key**                            | **Migration**                                                                                                                                                                                         |
| :----------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `console.overrideConfiguration`            | Use `console.extraConfiguration` (array format) instead.                                                                                                                                              |
| `console.tls.existingSecret`               | Use `console.tls.secret.existingSecret` instead.                                                                                                                                                      |
| `global.elasticsearch.tls.existingSecret`  | Use `global.elasticsearch.tls.secret.existingSecret` instead.                                                                                                                                         |
| `global.opensearch.tls.existingSecret`     | Use `global.opensearch.tls.secret.existingSecret` instead.                                                                                                                                            |
| `orchestration.profiles.identity`          | Renamed to `orchestration.profiles.admin`.                                                                                                                                                            |
| `webModeler.restapi.externalDatabase.user` | Renamed to `webModeler.restapi.externalDatabase.username`.                                                                                                                                            |
| `identity.keycloak`                        | Renamed in 8.7 to `identityKeycloak`; the bundled subchart itself is also removed in 8.10 — see [Bundled Bitnami subcharts removed](#bundled-bitnami-subcharts-removed) below.                        |
| `identityKeycloak`                         | Bundled Bitnami Keycloak subchart removed. Migrate to an external Keycloak instance (for example, the [Keycloak Operator](/self-managed/deployment/helm/configure/operator-based-infrastructure.md)). |
| `identityPostgresql`                       | Bundled Bitnami PostgreSQL subchart for Identity removed. Migrate to an externally managed PostgreSQL instance.                                                                                       |
| `webModelerPostgresql`                     | Bundled Bitnami PostgreSQL subchart for Web Modeler removed. Migrate to an externally managed PostgreSQL instance.                                                                                    |
| `elasticsearch`                            | Bundled Bitnami Elasticsearch subchart removed. Migrate to a managed Elasticsearch or OpenSearch service.                                                                                             |

#### Bundled Bitnami subcharts removed

The bundled Bitnami subcharts that were deprecated in 8.9 are **removed in 8.10**. The 8.10 chart rejects the following top-level keys with a hard constraint error:

- `identityKeycloak`
- `identityPostgresql`
- `webModelerPostgresql`
- `elasticsearch`

If any of these are present in your `values.yaml` (whether the subchart is enabled or not), the `helm upgrade` command fails before any pod is changed.

You must migrate to externally managed infrastructure before upgrading. Recommended replacements:

- **PostgreSQL** (for Identity and Web Modeler): a managed database (AWS RDS, GCP Cloud SQL, Azure Database for PostgreSQL) or a self-hosted instance provisioned via the [CloudNativePG operator](/self-managed/deployment/helm/configure/operator-based-infrastructure.md#cloudnative-pg-postgresql).
- **Keycloak**: the [Keycloak Operator](/self-managed/deployment/helm/configure/operator-based-infrastructure.md#keycloak-operator) or an external Keycloak instance.
- **Elasticsearch** (or **OpenSearch**): a managed Elasticsearch / OpenSearch service, or a self-hosted cluster provisioned via the [Elastic Cloud on Kubernetes operator](/self-managed/deployment/helm/configure/operator-based-infrastructure.md).

After standing up the external services, point the chart at them via the existing external-connection keys (`global.elasticsearch.external`, `identity.externalDatabase`, `webModeler.restapi.externalDatabase`, `global.identity.keycloak.url`, and friends). Migrate data from the bundled instances **before** you remove the subchart entries from `values.yaml`.

:::warning Data migration is destructive if skipped
Removing the Bitnami subchart keys deletes the corresponding `StatefulSet`s and their `PersistentVolumeClaim`s on the next reconcile. Back up and migrate data to the external services first, then update `values.yaml`, then run the upgrade.
:::

## Run the upgrade

After updating `values-8.10.yaml`, run the Helm upgrade:

```bash
helm repo update
helm upgrade <RELEASE_NAME> camunda/camunda-platform \
  --version <CHART_VERSION> \
  --namespace <NAMESPACE> \
  -f values-8.10.yaml
```

Replace `<RELEASE_NAME>`, `<CHART_VERSION>`, and `<NAMESPACE>` with your deployment-specific values.

:::note External infrastructure required
The 8.10 chart no longer ships bundled Keycloak, PostgreSQL, or Elasticsearch subcharts. Ensure your `values.yaml` points at externally managed services before running the upgrade — see [Bundled Bitnami subcharts removed](#bundled-bitnami-subcharts-removed).
:::

### Watch pod rollout progress

Watch the pods being upgraded via:

```bash
kubectl -n <NAMESPACE> get pods -w
```

You should see pods terminating and restarting with updated images.

### Validate the upgrade

1. Confirm all pods are `Running` and `Ready`:

   ```bash
   kubectl -n <NAMESPACE> get pods
   ```

2. Confirm every container is running an `8.10.x` image:

   ```bash
   kubectl -n <NAMESPACE> get pods \
     -o custom-columns=POD:.metadata.name,IMAGE:.spec.containers[*].image
   ```

3. Smoke-test the Orchestration Cluster REST API:

   ```bash
   kubectl -n <NAMESPACE> exec deploy/<RELEASE_NAME>-orchestration -- \
     curl -sf http://localhost:8080/actuator/health | head
   ```

4. Sign in to Console and Web Modeler in your browser. Confirm the Console UI loads under the Camunda Hub navigation, and that an existing Web Modeler project opens without errors.

5. Trigger a job from an existing worker (or your CI smoke suite) and confirm completion. See [Post-installation checks](/self-managed/operational-guides/troubleshooting/troubleshoot-helm-deployment.md) for additional checks.

## Troubleshooting

### Upgrade failed due to removed key

If your upgrade fails with an error like:

```
[camunda][error] The Helm values file key "console.overrideConfiguration" has been removed.
For more details, please check Camunda Helm chart documentation.
```

Remove the listed key from your `values.yaml` and retry the upgrade. See [Removed keys](#removed-keys) for the full list and their replacements.

### Upgrade failed due to missing secrets

If your upgrade fails due to missing credentials (often when using Bitnami subcharts), see [Extract plaintext values and reference them as Kubernetes Secrets](/self-managed/deployment/helm/configure/secret-management.md#extract-plaintext-values-and-reference-them-as-kubernetes-secrets).

### Web Modeler image pull errors after upgrade

If Web Modeler pods show `ImagePullBackOff` after upgrade, check whether your `values.yaml` pins image repositories to the old names (`camunda/web-modeler-restapi`, `camunda/web-modeler-websockets`). Update them to the new names (`camunda/hub`, `camunda/hub-websockets`). See [Image repository changes](#image-repository-changes).

### Console UI does not appear after upgrade

If the Web Modeler pod is healthy but the Console tab is missing from the navigation, confirm that either `camundaHub.enabled: true` or the legacy `console.enabled: true` is set in your `values.yaml`. The Console UI is gated on these keys — without one of them, the `hub-restapi` container starts with the Console feature flag disabled and the UI is hidden.

You can verify the wiring by inspecting the env on the running pod:

```bash
kubectl -n <NAMESPACE> get pod -l app.kubernetes.io/component=web-modeler-restapi \
  -o jsonpath='{.items[0].spec.containers[0].env[?(@.name=="CAMUNDA_MODELER_FEATURE_CONSOLE_ENABLED")].value}'
```

The expected value is `"true"`.
